# 🎛 Branding Intake Policy & SOP

## 🌌 Overview
本ワークフローは、Convoy 傘下の各プロダクトに独自の個性を与えつつ、全体の「Mission Control」としての統一感を維持するための「儀式」です。

ここで確定し `assets/branding/<productId>/brief.md` に保存される内容は、以後の画像生成、コード実装、およびドキュメント執筆における**唯一の正本（Source of Truth）**として機能します。

## ⚖️ Rules / Constraints
エージェントは、ブランディングの策定において以下の制約を「憲法」として厳守しなければなりません。

- **正本の所在地**: 必ず `assets/branding/<productId>/brief.md` を作成し、これをプロジェクトの最上位指針とすること。
- **言語の正本**: 思考、出力、およびドキュメントはすべて **日本語を正** とする。英語の使用は UI 語彙や固有名詞、技術用語に限定すること。
- **知的財産の保護**: 特定の著作権作品、既存のキャラクター、またはプロダクトの本質から外れる表現（例：「和風・高級」など）を厳禁とする。
- **運用の透明性**: ブランド決定のプロセスは必ず `walkthrough.md` に記録し、エージェントの独断で仕様を確定させてはならない。

## 🚀 Workflow / SOP
エージェントは以下のステップに従い、Commander（ユーザー）との対話を通じてブランドを確立します。

### Step 1: 収集（Intake Questions）
以下の項目を順に質問し、ユーザーの意図を正確に回収します。
1. **核**: アプリ名と、その存在意義を示す 1 文のコンセプト。
2. **対象**: 想定されるターゲット層と、具体的な利用シーン。
3. **基盤**: ターゲットプラットフォーム（Web/iOS/Android 等）と開発における最優先事項。
4. **感性**: UI の雰囲気を表すキーワード（最大 3 つ）、色・質感の希望、抽象的なモチーフ。
5. **規律**: 望ましい言葉遣いのトーン、および絶対に避けたい NG 表現。

### Step 2: 合意（Consensus）
回収した回答を「決定事項リスト」として日本語で要約し、ユーザーに提示します。ユーザーからの修正または承認を得るまで、次の成果物生成プロセスへ進んではなりません。

### Step 3: 結晶化（Artifact Generation）
承認された内容に基づき、以下の資産を生成・配置します。
1. **brief.md**: 確定したブランド指針を `assets/branding/<productId>/` に保存する。
2. **header_prompt.txt**: `brief.md` から派生させ、DALL-E 等で利用可能な画像生成用プロンプトを出力する。
3. **完了報告**: 生成ファイルと、次に行うべきアクション（例：`/generate-header-image` の提案）を日本語で報告する。

## ✅ Checklist
エージェントは最終出力の前に、以下の項目が満たされているか確認してください。

- [ ] `productId` は `workspace.config.json` やディレクトリ命名規則と整合しているか？
- [ ] 質問項目は日本語で、ユーザーが直感的に回答しやすい形式になっているか？
- [ ] `brief.md` に「特定作品を想起させない」という安全ガードが含まれているか？
- [ ] 最終報告が簡素になりすぎておらず、必要な導線（次のコマンド等）が含まれているか？
